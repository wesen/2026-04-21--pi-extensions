# Intern Implementation Guide: Metadata-Rich Response Viewer Markdown

## 1. Executive summary

The goal of this ticket is to make the Markdown generated by Response Viewer self-orienting. Today, when a user runs `/response-view`, `/rv`, or `/rv-last`, the extension saves an assistant response to a temporary Markdown file and opens it with `md-view`. The generated file has a small YAML header and then the response body. That is readable, but it does not tell the user enough about where the response came from or which documents shaped it.

This guide explains how to extend Response Viewer so each generated Markdown document contains two metadata layers:

- A **YAML frontmatter layer** for tools and future automation. This layer should use **absolute paths** for any file path fields.
- A **human-readable introduction layer** near the beginning of the Markdown file. This layer should use **relative Markdown links** so the document is easy to read in `md-view`.

The finished output should let a reader answer these questions immediately:

- Which Pi session did this response come from?
- Which assistant turn is this?
- Which history entry produced the response?
- Which model produced it?
- Where was the generated response Markdown saved?
- Which Markdown/documents were generated, edited, or read in the previous turn?
- How do I click from this response document to those related documents?

## 2. Vocabulary and mental model

Before touching code, learn these terms. They appear throughout this repository and in Pi extension APIs.

### 2.1 Pi extension

A Pi extension is a TypeScript module loaded by the Pi coding-agent runtime. An extension can register slash commands, actions, settings, docs, dashboard widgets, tools, and event handlers.

In this repository, every extension should call `registerPiExtension()` from:

- `extensions/_shared/registry.ts`

Response Viewer follows that convention in:

- `extensions/response-viewer/index.ts`

### 2.2 Response Viewer

Response Viewer is the extension being changed. Its user-facing job is:

1. Find assistant responses from the current Pi session branch.
2. Let the user browse/select a response, or choose the latest response directly.
3. Save that response as Markdown under the temp directory.
4. Open the Markdown with `md-view`.

Important files:

- `extensions/response-viewer/index.ts` — registration and command wiring.
- `extensions/response-viewer/response.ts` — response extraction, Markdown rendering, file writing, and md-view invocation.
- `extensions/response-viewer/ui.ts` — interactive response picker.
- `extensions/response-viewer/README.md` — current user-facing docs.

### 2.3 md-view

`md-view` is an external CLI that renders Markdown in a browser. Response Viewer invokes it through `pi.exec()`:

```ts
await pi.exec("md-view", args, { cwd: ctx.cwd, timeout: 15_000 });
```

The current command shape is:

```text
md-view view [--dark] [--no-reload] [--browser <cmd>] <path>
```

Response Viewer writes files first, then passes the generated file path to `md-view view`.

### 2.4 Pi session history

Pi keeps a session history containing entries. The current branch is retrieved with:

```ts
ctx.sessionManager.getBranch()
```

Response Viewer already uses this to reconstruct assistant responses. That matters because it survives `/reload`: instead of only remembering responses captured during this extension instance, it recomputes them from session history.

The rough session shape used by existing code is:

```ts
for (const entry of ctx.sessionManager.getBranch()) {
  if (entry.type !== "message") continue;
  const message = entry.message;

  if (message.role === "assistant") {
    // Assistant text and tool-call blocks live here.
  }

  if (message.role === "toolResult") {
    // Tool execution results live here.
  }
}
```

The code currently uses `any` in a few places because the exact session entry shape is runtime-owned by Pi.

### 2.5 Previous-turn document context

The user asked for links to documents generated or read in the previous turn. For this ticket, define “previous turn” precisely:

> For a selected assistant response, previous-turn document context means successful relevant tool results that occurred after the previous assistant response and before the selected assistant response.

This definition works well because the selected assistant response is usually a narrative summary of tool calls that happened immediately before it. Those tool calls are the files the response was based on.

Example sequence:

```text
assistant response A
user asks for docs
assistant toolCall: read docs/pi-testing-guide.md
toolResult: read OK
assistant toolCall: write ttmp/.../guide.md
toolResult: write OK
assistant response B
```

When rendering response B, the previous-turn document context is:

- `docs/pi-testing-guide.md` as a read document.
- `ttmp/.../guide.md` as a generated document.

## 3. Current system architecture

This section explains what exists today before the ticket is implemented.

### 3.1 High-level flow today

```text
User runs /response-view or /rv-last
          |
          v
extensions/response-viewer/index.ts
          |
          | calls getResponsesFromSession(ctx)
          v
extensions/response-viewer/response.ts
          |
          | returns CapturedResponse[]
          v
Picker or last-response selector
          |
          | selected CapturedResponse
          v
saveToTempFile(response)
          |
          | writes /tmp/pi-response-viewer/last-response.md
          | writes /tmp/pi-response-viewer/<timestamp>-turn-N.md
          v
openWithMdView(pi, ctx, state, path)
          |
          | runs md-view view <path>
          v
Browser opens rendered Markdown
```

The code paths are easy to follow:

- `/rv` and `/response-view` call `openPicker()`.
- `/rv-last` calls `saveAndOpenLast()`.
- Both eventually call `saveAndOpenResponse()`.
- `saveAndOpenResponse()` calls `saveToTempFile(response)`.
- `saveToTempFile()` calls `renderMarkdown(response)`.
- `openWithMdView()` opens the saved file.

### 3.2 Extension registration

`extensions/response-viewer/index.ts` exports the extension default function:

```ts

    actions: [...],

    widgets: [...],
    palette: [...],
  });

  pi.on("turn_end", async (event, ctx) => { ... });
  pi.registerCommand("rv", { ... });
  pi.registerCommand("response-view", { ... });
  pi.registerCommand("rv-last", { ... });
  pi.registerCommand("rv-preview", { ... });
  pi.registerCommand("rv-reopen", { ... });

```

Important concepts from `extensions/_shared/registry.ts`:

- `id` is the stable machine identifier.
- `name` and `description` are user-facing strings.
- `run` is the default launcher action.
- `actions` are named verbs shown in `/px`.
- `docs` point to help pages.
- `settings` provide a schema-driven settings UI.
- `widgets` render status/dashboard text.
- `palette` contributes keyboard-driven command-palette items.

You should not bypass this registry. Keep the existing registration and add metadata behavior under the existing response-viewer architecture.

### 3.3 CapturedResponse today

`extensions/response-viewer/response.ts` defines:

```ts
export interface CapturedResponse {
  turnIndex: number;
  capturedAt: string;
  sessionId: string;
  entryId: string;
  modelProvider: string | undefined;
  modelId: string | undefined;
  modelName: string | undefined;
  text: string;
  textLength: number;
}
```

This already contains most of the session metadata required by the ticket:

- `sessionId` — from `ctx.sessionManager.getSessionId()`.
- `entryId` — from the session history entry.
- `turnIndex` — zero-based assistant-response index within extracted responses.
- `turnIndex + 1` — human-facing turn number.
- `capturedAt` — timestamp of the history entry.
- model fields — from `ctx.model`.
- `text` and `textLength` — rendered response content.

### 3.4 Response extraction today

Current extraction logic:

```ts
export function getResponsesFromSession(ctx: ExtensionContext): CapturedResponse[] {
  const entries = ctx.sessionManager.getBranch();
  const sessionId = ctx.sessionManager.getSessionId();
  const responses: CapturedResponse[] = [];
  let turnIndex = 0;

  for (const entry of entries) {
    if (entry.type !== "message") continue;
    const message = (entry as any).message;
    if (!message || message.role !== "assistant") continue;
    if (!Array.isArray(message.content)) continue;

    const text = extractTextFromContent(message.content);
    if (!text) continue;

    responses.push({
      turnIndex,
      capturedAt: entry.timestamp,
      sessionId,
      entryId: entry.id,
      modelProvider: ctx.model?.provider,
      modelId: ctx.model?.id,
      modelName: ctx.model?.name,
      text,
      textLength: text.length,
    });
    turnIndex++;
  }

  return responses;
}
```

Two details matter:

- It uses `ctx.sessionManager.getBranch()`, not all entries. This respects the active conversation branch.
- It strips `<summary>...</summary>` blocks from assistant text before saving and previewing.

### 3.5 Markdown rendering today

Current renderer:

```ts
export function renderMarkdown(response: CapturedResponse): string {
  return [
    "---",
    `Title: ${yamlString(`Pi Response — Turn ${response.turnIndex + 1}`)}`,
    `Source: ${yamlString("pi-response-viewer")}`,
    `SessionId: ${yamlString(response.sessionId)}`,
    `TurnIndex: ${response.turnIndex}`,
    `CapturedAt: ${yamlString(response.capturedAt)}`,
    `ModelProvider: ${yamlString(response.modelProvider)}`,
    `ModelId: ${yamlString(response.modelId)}`,
    `ModelName: ${yamlString(response.modelName)}`,
    "---",
    "",
    `# Pi Response — Turn ${response.turnIndex + 1}`,
    "",
    response.text,
    "",
  ].join("\n");
}
```

This is the main function to change. However, do not simply append metadata here without thinking about paths. The renderer currently does not know where the generated Markdown file will live. For correct relative links, it needs either:

- the output path, or
- a precomputed render context containing relative links from that output path.

### 3.6 Temp-file writing today

Current writer:

```ts
export function saveToTempFile(response: CapturedResponse, overrideDir?: string): string {
  const dir = ensureTempDir(overrideDir);

  const lastPath = join(dir, "last-response.md");
  writeFileSync(lastPath, renderMarkdown(response), "utf-8");

  const slug = timestampSlug();
  const timestampedPath = join(dir, `${slug}-turn-${response.turnIndex + 1}.md`);
  writeFileSync(timestampedPath, renderMarkdown(response), "utf-8");

  return lastPath;
}
```

Important behavior to preserve:

- `last-response.md` is always overwritten.
- Timestamped copies are kept for history.
- The function returns the `last-response.md` path because `md-view` live-reloads that stable path.

This ticket likely needs to modify `saveToTempFile()` so it computes both output paths before rendering. That lets frontmatter include output paths and lets the body links be relative to the file being written.

## 4. Desired output format

This section describes what the generated Markdown should look like after implementation.

### 4.1 Frontmatter rules

The frontmatter is for machines. It should be stable, explicit, and unambiguous.

Rules:

- Use YAML frontmatter bounded by `---`.
- Use absolute paths for all path fields.
- Prefer lower-case stable keys for new fields.
- It is acceptable to keep old keys if you want compatibility, but do not rely on old keys for the new design.
- Escape strings safely.
- Represent missing values as empty strings or `null`, but be consistent.

Target shape:

```yaml
---
title: "Pi Response — Turn 12"
source: "pi-response-viewer"
session:
  id: "session-id"
  responseEntryId: "entry-id"
  turnIndex: 11
  turnNumber: 12
capturedAt: "2026-05-29T12:34:56.000Z"
model:
  provider: "anthropic"
  id: "claude-example"
  name: "Claude Example"
paths:
  lastResponse: "/tmp/pi-response-viewer/last-response.md"
  timestampedCopy: "/tmp/pi-response-viewer/2026-05-29T12-34-56-000Z-turn-12.md"
documents:
  generated:
    - path: "/home/manuel/code/wesen/2026-04-21--pi-extensions/ttmp/.../guide.md"
      relativePath: "ttmp/.../guide.md"
      linkTarget: "../../home/manuel/.../guide.md"
      toolName: "write"
      toolCallId: "tool-call-id"
      entryId: "tool-result-entry-id"
      exists: true
  read:
    - path: "/home/manuel/code/wesen/2026-04-21--pi-extensions/docs/pi-testing-guide.md"
      relativePath: "docs/pi-testing-guide.md"
      linkTarget: "../../home/manuel/.../docs/pi-testing-guide.md"
      toolName: "read"
      toolCallId: "tool-call-id"
      entryId: "tool-result-entry-id"
      exists: true
---
```

`path` must be absolute. `relativePath` is useful as a display label, but it is not a substitute for `path`.

### 4.2 Human-readable introduction rules

The introduction is for people. It should be short, scannable, and placed before the response text.

Target shape:

```markdown
# Pi Response — Turn 12

> Session `session-id`, turn 12, captured `2026-05-29T12:34:56.000Z`.
> Model: `anthropic/claude-example`.
> Previous-turn context: 1 generated document, 1 read document.

## Context metadata

- **Session:** `session-id`
- **Entry:** `entry-id`
- **Turn:** 12 (index 11)
- **Captured:** `2026-05-29T12:34:56.000Z`
- **Model:** `anthropic/claude-example` (`Claude Example`)
- **Saved files:**
  - `last-response.md`: `/tmp/pi-response-viewer/last-response.md`
  - timestamped copy: `/tmp/pi-response-viewer/2026-...-turn-12.md`

### Generated documents from previous turn

- [ttmp/.../guide.md](relative/link/from/generated/file/to/guide.md) — `write`, exists

### Documents read in previous turn

- [docs/pi-testing-guide.md](relative/link/from/generated/file/to/docs/pi-testing-guide.md) — `read`, exists

---

## Response

<assistant response body>
```

Rules:

- Use relative Markdown links in link targets.
- Use cwd-relative labels where possible, because they are readable.
- If a file is missing, do not create a broken link without warning. Either render plain code text with “missing” or render a link with a clear missing marker.
- Keep the intro short enough that it helps orientation without burying the actual response.

## 5. Data design

### 5.1 New types

Add explicit types near the top of `extensions/response-viewer/response.ts`. You can refine names during implementation, but keep the separation between response data, document context, and render paths.

Recommended type sketch:

```ts
type ResponseDocumentKind = "generated" | "read";
type ResponseDocumentToolName = "write" | "edit" | "read";

export interface ResponseDocumentContextItem {
  kind: ResponseDocumentKind;
  toolName: ResponseDocumentToolName;
  toolCallId: string;
  entryId: string;
  absolutePath: string;
  displayPath: string;      // cwd-relative when possible
  linkTarget: string;       // relative from generated markdown file
  exists: boolean;
  timestamp: string | undefined;
}

export interface ResponseOutputPaths {
  lastResponsePath: string;
  timestampedPath: string;
}

export interface ResponseMarkdownContext {
  title: string;
  source: "pi-response-viewer";
  cwd: string;
  outputPath: string;
  outputPaths: ResponseOutputPaths;
  documents: ResponseDocumentContextItem[];
}
```

Why separate these?

- `CapturedResponse` is about the assistant response itself.
- `ResponseDocumentContextItem` is about nearby file activity in session history.
- `ResponseOutputPaths` is about where this render is being saved.
- `ResponseMarkdownContext` gives the renderer everything it needs without making it reach back into `ctx`.

### 5.2 Generated vs read documents

Use `kind` to classify documents:

- `generated` for successful `write` and `edit` tool results.
- `read` for successful `read` tool results.

If a file is both read and generated in the same previous-turn window, choose one of these approaches:

1. **Simple approach:** keep two entries, one in generated and one in read.
2. **Merged approach:** merge by absolute path and store `kinds: ["read", "generated"]`.

For an intern implementation, the simple approach is easier and acceptable.

### 5.3 Path fields

Each document context item should have these path forms:

| Field | Example | Purpose |
| --- | --- | --- |
| `absolutePath` | `/home/manuel/code/.../docs/foo.md` | YAML machine-readable identity |
| `displayPath` | `docs/foo.md` | Human-readable label |
| `linkTarget` | `../../home/manuel/code/.../docs/foo.md` or repo-relative route if output is inside repo | Markdown link target relative to generated file |

The tricky field is `linkTarget`. It must be relative to the output Markdown file directory, not relative to `ctx.cwd`.

Pseudocode:

```ts
import { dirname, relative } from "node:path";

function markdownLinkTarget(fromMarkdownPath: string, targetAbsolutePath: string): string {
  const fromDir = dirname(fromMarkdownPath);
  const rel = relative(fromDir, targetAbsolutePath);
  return rel || ".";
}
```

Because links can contain spaces or special characters, also escape link targets for Markdown:

```ts
function markdownLink(label: string, target: string): string {
  return `[${escapeMarkdownLinkLabel(label)}](${encodeURI(target)})`;
}
```

Do not over-engineer on the first pass. At minimum, handle parentheses and spaces well enough for local paths used in this repository.

## 6. Previous-turn document discovery design

This is the most important new logic.

### 6.1 Existing pattern to copy from

`extensions/markdown-recent-viewer/history.ts` already solves a similar problem:

- Walk session entries.
- Find assistant `toolCall` blocks.
- Remember pending tool calls by id.
- Match later `toolResult` messages by `toolCallId`.
- Ignore failed results.
- Resolve paths relative to `ctx.cwd`.
- Deduplicate.

Relevant existing functions and ideas:

```ts
function isToolCallBlock(block: unknown): block is {
  type: "toolCall";
  id: string;
  name: string;
  arguments: Record<string, unknown>;
}

function isMarkdownToolName(name: string): name is MarkdownToolName {
  return name === "edit" || name === "write";
}

const absolutePath = resolve(ctx.cwd, rawPath);
```

For this ticket, extend the concept to include `read`.

### 6.2 The session window

For a selected response, find the relevant window:

```text
entries: [ ... previous assistant response ... tool calls/results ... selected assistant response ... ]
                                       ^ window begins              ^ window ends
```

The easiest robust algorithm is:

1. Get `entries = ctx.sessionManager.getBranch()`.
2. Find `selectedIndex`, the index where `entry.id === response.entryId`.
3. Scan backward from `selectedIndex - 1` until you find the previous assistant message with text. The window starts after that previous assistant message.
4. Scan forward from `windowStart` up to but not including `selectedIndex`.
5. Within that window, pair tool calls with tool results.

Pseudocode:

```ts
function getPreviousTurnWindow(entries, responseEntryId): SessionEntry[] {
  const selectedIndex = entries.findIndex((entry) => entry.id === responseEntryId);
  if (selectedIndex < 0) return [];

  let start = 0;
  for (let i = selectedIndex - 1; i >= 0; i--) {
    const entry = entries[i];
    if (isAssistantMessageEntry(entry) && hasTextContent(entry.message)) {
      start = i + 1;
      break;
    }
  }

  return entries.slice(start, selectedIndex);
}
```

Why look for assistant text? Because assistant entries can contain tool calls as well as final response text. You want the previous completed assistant response, not necessarily an assistant tool-call block inside the current turn.

If the runtime structure makes this distinction hard, start with “previous message entry with `message.role === "assistant"` and non-empty extracted text.” That matches existing response extraction logic.

### 6.3 Pairing tool calls and results

Tool call information lives in assistant message content blocks. Tool result information lives in tool-result messages. The path usually comes from the tool call arguments.

Pseudocode:

```ts
interface PendingDocumentToolCall {
  id: string;
  name: "read" | "write" | "edit";
  rawPath: string;
  absolutePath: string;
}

function collectDocumentsFromWindow(ctx, entries, outputPath): ResponseDocumentContextItem[] {
  const pendingById = new Map<string, PendingDocumentToolCall>();
  const items: ResponseDocumentContextItem[] = [];

  for (const entry of entries) {
    if (entry.type !== "message") continue;
    const message = entry.message as any;

    if (message.role === "assistant" && Array.isArray(message.content)) {
      for (const block of message.content) {
        if (!isToolCallBlock(block)) continue;
        if (!isDocumentToolName(block.name)) continue;

        const rawPath = block.arguments.path;
        if (typeof rawPath !== "string" || !rawPath.trim()) continue;

        const absolutePath = resolve(ctx.cwd, rawPath);
        if (!isDocumentPath(absolutePath)) continue;

        pendingById.set(block.id, {
          id: block.id,
          name: block.name,
          rawPath,
          absolutePath,
        });
      }
    }

    if (message.role === "toolResult") {
      if (message.isError === true) continue;
      const pending = pendingById.get(message.toolCallId);
      if (!pending) continue;

      items.push(toDocumentContextItem(ctx, entry, message, pending, outputPath));
    }
  }

  return dedupeDocumentItems(items);
}
```

### 6.4 Tool names and path extraction

Start with these tool names:

- `read` — read a file.
- `write` — create or overwrite a file.
- `edit` — edit an existing file.

Expected argument shape:

```ts
block.arguments.path
```

Be defensive:

- Only use the path if it is a non-empty string.
- Resolve relative paths against `ctx.cwd`.
- Filter to document-like extensions for first implementation.

Recommended extension filter:

```ts
const DOCUMENT_EXTENSIONS = new Set([".md", ".markdown", ".mdx", ".txt"]);
```

The user specifically asked for Markdown documents, so `.md` and `.markdown` are the core. `.txt` can be included if useful, but avoid sweeping in source files unless the design is expanded.

### 6.5 Deduplication

Do not show the same generated file five times if it was edited repeatedly in the same turn. A good first-pass rule:

- Key by `${kind}:${absolutePath}`.
- Keep the latest occurrence in the window.

Pseudocode:

```ts
function dedupeDocumentItems(items: ResponseDocumentContextItem[]): ResponseDocumentContextItem[] {
  const latestByKey = new Map<string, ResponseDocumentContextItem>();
  for (const item of items) {
    latestByKey.set(`${item.kind}:${item.absolutePath}`, item);
  }
  return [...latestByKey.values()];
}
```

### 6.6 Missing files

Files may disappear after the response was generated. This matters because `/response-view` can render old responses from session history.

Use `existsSync(absolutePath)` when building context.

Rendering rule:

- If `exists === true`, render a clickable Markdown link.
- If `exists === false`, render the display path in code ticks plus `(missing)`.

Example:

```markdown
- `ttmp/old/deleted.md` — `write`, missing
```

## 7. Rendering design

### 7.1 Refactor saveToTempFile first

Current `saveToTempFile(response)` calls `renderMarkdown(response)` before the renderer knows the output paths. Change it to compute paths first.

Proposed signature:

```ts
export function saveToTempFile(
  ctx: ExtensionContext,
  response: CapturedResponse,
  overrideDir?: string,
): string
```

However, changing this signature affects callers in `index.ts`. That is acceptable if you update all call sites. Existing callers have `ctx` available.

Alternative if you want a smaller call-site change:

```ts
export function saveToTempFile(
  response: CapturedResponse,
  options?: { ctx?: ExtensionContext; overrideDir?: string },
): string
```

The clearer implementation is the first one: pass `ctx` explicitly.

Proposed flow:

```ts
export function saveToTempFile(ctx: ExtensionContext, response: CapturedResponse, overrideDir?: string): string {
  const dir = ensureTempDir(overrideDir);
  const lastPath = join(dir, "last-response.md");
  const slug = timestampSlug();
  const timestampedPath = join(dir, `${slug}-turn-${response.turnIndex + 1}.md`);

  const baseOutputPaths = { lastResponsePath: lastPath, timestampedPath };

  const lastMarkdown = renderMarkdown(response, buildRenderContext(ctx, response, lastPath, baseOutputPaths));
  writeFileSync(lastPath, lastMarkdown, "utf-8");

  const copyMarkdown = renderMarkdown(response, buildRenderContext(ctx, response, timestampedPath, baseOutputPaths));
  writeFileSync(timestampedPath, copyMarkdown, "utf-8");

  return lastPath;
}
```

Why render twice? Because relative links differ depending on the output file location. Today both files are in the same directory, so the links will usually be the same, but rendering separately is correct and future-proof.

### 7.2 Build a render context

Pseudocode:

```ts
function buildRenderContext(
  ctx: ExtensionContext,
  response: CapturedResponse,
  outputPath: string,
  outputPaths: ResponseOutputPaths,
): ResponseMarkdownContext {
  return {
    title: `Pi Response — Turn ${response.turnIndex + 1}`,
    source: "pi-response-viewer",
    cwd: ctx.cwd,
    outputPath,
    outputPaths,
    documents: getPreviousTurnDocumentContext(ctx, response, outputPath),
  };
}
```

This context is the bridge between runtime state and pure rendering. Keeping it explicit makes `renderMarkdown()` easier to test.

### 7.3 YAML rendering

The current `yamlString()` helper handles one-line strings by using `JSON.stringify()`. For nested YAML, either:

1. Write a small explicit YAML renderer for the known shape.
2. Add/use a YAML library if already available.

Prefer option 1 unless the repository already depends on a YAML library. The shape is small enough to render manually.

Example helpers:

```ts
function yamlScalar(value: string | number | boolean | undefined): string {
  if (typeof value === "number") return String(value);
  if (typeof value === "boolean") return value ? "true" : "false";
  if (value === undefined || value === "") return '""';
  return JSON.stringify(value);
}

function indent(lines: string[], spaces: number): string[] {
  const prefix = " ".repeat(spaces);
  return lines.map((line) => `${prefix}${line}`);
}
```

Then render known sections explicitly:

```ts
function renderFrontmatter(response: CapturedResponse, meta: ResponseMarkdownContext): string[] {
  const generated = meta.documents.filter((d) => d.kind === "generated");
  const read = meta.documents.filter((d) => d.kind === "read");

  return [
    "---",
    `title: ${yamlScalar(meta.title)}`,
    `source: ${yamlScalar(meta.source)}`,
    "session:",
    `  id: ${yamlScalar(response.sessionId)}`,
    `  responseEntryId: ${yamlScalar(response.entryId)}`,
    `  turnIndex: ${response.turnIndex}`,
    `  turnNumber: ${response.turnIndex + 1}`,
    `capturedAt: ${yamlScalar(response.capturedAt)}`,
    "model:",
    `  provider: ${yamlScalar(response.modelProvider)}`,
    `  id: ${yamlScalar(response.modelId)}`,
    `  name: ${yamlScalar(response.modelName)}`,
    "paths:",
    `  lastResponse: ${yamlScalar(meta.outputPaths.lastResponsePath)}`,
    `  timestampedCopy: ${yamlScalar(meta.outputPaths.timestampedPath)}`,
    "documents:",
    "  generated:",
    ...renderYamlDocumentList(generated, 4),
    "  read:",
    ...renderYamlDocumentList(read, 4),
    "---",
  ];
}
```

Empty list rendering should be valid YAML:

```yaml
documents:
  generated: []
  read: []
```

That may be cleaner than emitting a key followed by no items.

### 7.4 Body rendering

Keep the body renderer separate from the YAML renderer.

Pseudocode:

```ts
function renderIntro(response: CapturedResponse, meta: ResponseMarkdownContext): string[] {
  const generated = meta.documents.filter((d) => d.kind === "generated");
  const read = meta.documents.filter((d) => d.kind === "read");

  return [
    `# ${meta.title}`,
    "",
    `> Session \`${response.sessionId}\`, turn ${response.turnIndex + 1}, captured \`${response.capturedAt}\`.`,
    `> Model: \`${formatModel(response)}\`.`,
    `> Previous-turn context: ${generated.length} generated document(s), ${read.length} read document(s).`,

    "## Context metadata",
    "",
    `- **Session:** \`${response.sessionId}\``,
    `- **Entry:** \`${response.entryId}\``,
    `- **Turn:** ${response.turnIndex + 1} (index ${response.turnIndex})`,
    `- **Captured:** \`${response.capturedAt}\``,
    `- **Model:** \`${formatModel(response)}\`${response.modelName ? ` (\`${response.modelName}\`)` : ""}`,
    "- **Saved files:**",
    `  - \`last-response.md\`: \`${meta.outputPaths.lastResponsePath}\``,
    `  - timestamped copy: \`${meta.outputPaths.timestampedPath}\``,
    "",
    "### Generated documents from previous turn",
    "",
    ...renderMarkdownDocumentList(generated),
    "",
    "### Documents read in previous turn",
    "",
    ...renderMarkdownDocumentList(read),
    "",
    "---",
    "",
    "## Response",
    "",
  ];
}
```

### 7.5 Full renderer

After splitting helpers, `renderMarkdown()` becomes a coordinator:

```ts
export function renderMarkdown(response: CapturedResponse, meta: ResponseMarkdownContext): string {
  return [
    ...renderFrontmatter(response, meta),
    "",
    ...renderIntro(response, meta),
    response.text,
    "",
  ].join("\n");
}
```

This is easier to review and test than one giant array literal.

## 10. Edge cases and design decisions

### 10.1 No previous-turn documents

The response might not be based on file reads/writes. Render this gracefully.

Frontmatter:

```yaml
documents:
  generated: []
  read: []
```

Body:

```markdown
### Generated documents from previous turn

- None detected.

### Documents read in previous turn

- None detected.
```

### 10.2 Tool result exists but pending tool call was not found

Skip it. Without the tool-call arguments, you usually cannot know the path reliably.

### 10.3 Tool call path is not a string

Skip it. Be defensive and avoid throwing during response rendering.

### 10.4 Missing files

Record them with `exists: false` in frontmatter. In the Markdown body, avoid pretending the link works.

### 10.5 Absolute paths in frontmatter may expose local paths

That is explicitly requested. Do not convert frontmatter paths to relative paths.

### 10.6 Relative links from `/tmp`

The generated response file lives under `/tmp/pi-response-viewer`, while repository files live elsewhere. A relative link from `/tmp/pi-response-viewer/last-response.md` to a repo file can look long and awkward, for example:

```text
../../home/manuel/code/wesen/2026-04-21--pi-extensions/docs/pi-testing-guide.md
```

That is still a relative link. It satisfies the requirement and should work as long as `md-view` resolves links relative to the opened Markdown file.

### 10.7 Branching/forking

Response Viewer uses `getBranch()`. That means it should render metadata for the active branch, not necessarily every historical fork. Preserve that behavior unless a future ticket asks otherwise.

### 10.8 Auto-open

Auto-open uses the same save path. Make sure metadata rendering works both when the user runs `/rv-last` manually and when `autoOpen` is enabled.

## 11. Suggested concrete code skeleton

This is not drop-in final code, but it shows the shape to implement.

```ts
import { existsSync, mkdirSync, writeFileSync } from "node:fs";
import { tmpdir } from "node:os";
import { dirname, extname, join, relative, resolve } from "node:path";

const DOCUMENT_EXTENSIONS = new Set([".md", ".markdown", ".mdx"]);

type ResponseDocumentKind = "generated" | "read";
type ResponseDocumentToolName = "write" | "edit" | "read";

interface ToolCallBlock {
  type: "toolCall";
  id: string;
  name: string;
  arguments: Record<string, unknown>;
}

export interface ResponseDocumentContextItem {
  kind: ResponseDocumentKind;
  toolName: ResponseDocumentToolName;
  toolCallId: string;
  entryId: string;
  absolutePath: string;
  displayPath: string;
  linkTarget: string;
  exists: boolean;
  timestamp: string | undefined;
}

export interface ResponseOutputPaths {
  lastResponsePath: string;
  timestampedPath: string;
}

export interface ResponseMarkdownContext {
  title: string;
  source: "pi-response-viewer";
  cwd: string;
  outputPath: string;
  outputPaths: ResponseOutputPaths;
  documents: ResponseDocumentContextItem[];
}

function isToolCallBlock(block: unknown): block is ToolCallBlock {
  if (!block || typeof block !== "object") return false;
  const candidate = block as Record<string, unknown>;
  return (
    candidate.type === "toolCall" &&
    typeof candidate.id === "string" &&
    typeof candidate.name === "string" &&
    !!candidate.arguments &&
    typeof candidate.arguments === "object"
  );
}

function isDocumentToolName(name: string): name is ResponseDocumentToolName {
  return name === "read" || name === "write" || name === "edit";
}

function isDocumentPath(path: string): boolean {
  return DOCUMENT_EXTENSIONS.has(extname(path).toLowerCase());
}

function documentKindForTool(toolName: ResponseDocumentToolName): ResponseDocumentKind {
  return toolName === "read" ? "read" : "generated";
}

function displayPath(cwd: string, absolutePath: string): string {
  const rel = relative(cwd, absolutePath);
  if (!rel || rel.startsWith("..") || rel === absolutePath) return absolutePath;
  return rel;
}

function makeLinkTarget(outputPath: string, absolutePath: string): string {
  return relative(dirname(outputPath), absolutePath) || ".";
}
```

The actual implementation will need to adapt to the exact runtime shape of session entries, but this is the intended structure.

## 14. What to avoid

Avoid these common mistakes:

- Do not compute Markdown link targets relative to `ctx.cwd`; compute them relative to the generated Markdown file path.
- Do not put only relative paths in frontmatter; the user explicitly requested absolute paths there.
- Do not scan all session history if the existing behavior uses the current branch. Use `getBranch()` unless there is a deliberate design change.
- Do not throw from rendering just because metadata extraction fails. If metadata cannot be collected, render the response with empty document lists and keep the command usable.
- Do not change `ResponsePicker` unless necessary. The UI picker does not need to understand document metadata.
- Do not remove the timestamped copy behavior.
- Do not make `md-view` invocation depend on metadata success.

## 15. Implementation north star

The best implementation keeps the core flow simple:

```text
Extract response from session history
  -> compute output paths
  -> discover previous-turn document context
  -> render YAML frontmatter with absolute paths
  -> render human intro with relative links
  -> append assistant response body
  -> write last-response.md and timestamped copy
  -> open last-response.md with md-view
```

If a proposed change does not support that flow, it probably belongs in a later ticket.
